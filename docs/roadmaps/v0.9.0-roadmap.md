# hf-fetch-model v0.9.0 Roadmap

**Date:** March 21, 2026
**Status:** Proposed
**Context:** v0.8.0 added cache-awareness (`list-files --show-cached`, `--dry-run` cache status). v0.8.1 improved partial download detection. v0.9.0 closes the loop: users can now *see* what's cached — now they need to *measure*, *verify*, and *prune* it.

---

## Goal

**Cache management** — three subcommands (`du`, `verify`, `clean`) and corresponding library functions for measuring, verifying, and pruning the local HuggingFace cache.

---

## Existing Infrastructure

v0.9.0 builds on code already in place:

| Module | What exists | What v0.9.0 adds |
|--------|------------|-------------------|
| `cache.rs` | `hf_cache_dir()`, `cache_summary()` → `Vec<CachedModelSummary>` (repo_id, file_count, total_size, has_partial), `repo_status()` → `RepoStatus` with per-file `FileStatus` (Complete/Partial/Missing), `read_ref()`, `count_snapshot_files()` | `du` reuses `cache_summary()` directly; `verify` reuses `repo_status()` + `checksum.rs`; `clean` adds deletion logic |
| `checksum.rs` | `verify_sha256(path, filename, expected_hex)` — async SHA256 verification | `verify` calls this per file against LFS metadata |
| `repo.rs` | `list_repo_files_with_metadata()` — fetches SHA256 hashes from HF API | `verify` fetches expected hashes via this function |
| `bin/main.rs` | `run_status()` / `run_status_all()` — display patterns for per-file and summary tables, `format_size()` | `du`, `verify`, `clean` follow the same display patterns |

## Subcommand 1: `hf-fm du [REPO_ID]`

### Why

`~/.cache/huggingface/hub/` grows unbounded. Users need to know what's taking space before they can decide what to prune.

### Behavior

Without arguments: all cached repos, sorted by size descending. With a repo: per-file breakdown. Repos with incomplete downloads show a `PARTIAL` marker.

```
$ hf-fm du
  5.10 GiB  google/gemma-2-2b-it          (8 files)
  2.80 GiB  EleutherAI/pythia-1.4b        (12 files)  PARTIAL
  1.20 GiB  google/gemma-scope-2b-pt-res  (3 files)
  ──────────────────────────────────────────────────
  9.10 GiB  total (3 repos, 23 files)

$ hf-fm du google/gemma-2-2b-it
  2.50 GiB  model-00001-of-00002.safetensors
  2.60 GiB  model-00002-of-00002.safetensors
  1.20 KiB  config.json
  ...
  ──────────────────────────────────────────────────
  5.10 GiB  total (8 files)
```

### Implementation

**Summary mode (no args):** Calls existing `cache::cache_summary()`, sorts by `total_size` descending, formats with `format_size()`. Almost zero new code.

**Per-repo mode:** Walks `<cache_dir>/models--<org>--<name>/snapshots/<commit>/` and lists each file with its size. Reuses `count_snapshot_files()` logic but returns per-file entries instead of totals.

### Library API

```rust
/// Per-file disk usage entry within a cached repository.
pub struct CacheFileUsage {
    /// Filename relative to the snapshot directory.
    pub filename: String,
    /// File size in bytes.
    pub size: u64,
}

/// Returns per-file disk usage for a specific cached repository.
///
/// # Errors
///
/// Returns [`FetchError::Io`] if the cache directory cannot be read.
pub fn cache_repo_usage(repo_id: &str) -> Result<Vec<CacheFileUsage>, FetchError>;
```

The summary-level API already exists as `cache_summary()`.

## Subcommand 2: `hf-fm verify <REPO_ID> [--revision <REV>] [--token <TOKEN>]`

### Why

Downloaded files can be corrupted (disk errors, interrupted writes, bit rot). After a download, `verify_checksums` runs automatically, but there's no way to re-verify existing cache entries later.

### Behavior

Re-verifies SHA256 checksums of cached files against HF LFS metadata. Requires network (to fetch expected hashes). `REPO_ID` is required — verifying all cached repos at once could take hours for large caches.

```
$ hf-fm verify google/gemma-2-2b-it
  ✓ model-00001-of-00002.safetensors  (2.50 GiB, SHA256 OK)
  ✓ model-00002-of-00002.safetensors  (2.60 GiB, SHA256 OK)
  — config.json                       (1.20 KiB, no LFS hash)
  3 files: 2 SHA256 OK, 1 skipped (no LFS metadata)
```

For corrupted files:

```
$ hf-fm verify google/gemma-2-2b-it
  ✓ model-00001-of-00002.safetensors  (2.50 GiB, SHA256 OK)
  ✗ model-00002-of-00002.safetensors  (2.60 GiB, SHA256 MISMATCH)
  — config.json                       (1.20 KiB, no LFS hash)
  3 files: 1 SHA256 OK, 1 FAILED, 1 skipped
```

### Implementation

1. Fetch expected hashes via `repo::list_repo_files_with_metadata(repo_id, token, revision)`
2. Resolve local paths via `cache::hf_cache_dir()` + `read_ref()` + snapshot directory
3. For each file with an LFS SHA256: call `checksum::verify_sha256(path, filename, expected_hex)`
4. Files without LFS metadata (small git-stored files): report "no LFS hash" — skip verification
5. Report summary: verified count, checked count, skipped count, failed count

### Library API

```rust
/// Result of verifying a single cached file.
pub struct FileVerification {
    /// Filename within the repository.
    pub filename: String,
    /// File size in bytes.
    pub size: u64,
    /// Verification result.
    pub status: VerifyStatus,
}

/// Verification status for a single file.
#[non_exhaustive]
pub enum VerifyStatus {
    /// SHA256 matches expected hash.
    Ok,
    /// SHA256 does not match (file is corrupted).
    Mismatch { expected: String, actual: String },
    /// File has no LFS metadata (small git-stored file); verification skipped.
    Skipped,
    /// File is missing from cache.
    Missing,
}

/// Verifies SHA256 checksums of cached files against HF LFS metadata.
///
/// # Errors
///
/// Returns [`FetchError::Http`] if the metadata fetch fails.
/// Returns [`FetchError::Io`] if cached files cannot be read.
pub async fn verify_cache(
    repo_id: &str,
    token: Option<&str>,
    revision: Option<&str>,
) -> Result<Vec<FileVerification>, FetchError>;
```

## Subcommand 3: `hf-fm clean <--older-than <DAYS> | --repo <REPO_ID> | --all> [--except <REPO_ID>] [--yes]`

### Why

Once users see disk usage (`du`) and know what's cached, they need a way to reclaim space. Manual deletion of `~/.cache/huggingface/hub/models--*` directories works but is error-prone (partial deletion, leftover refs).

### Behavior

Prune cached repos. **Requires at least one of `--older-than`, `--repo`, or `--all`** — running `clean` with no filter is an error (safety guard against accidental full deletion). Defaults to interactive confirmation. Removes the entire `models--<org>--<name>` directory (snapshots, blobs, refs).

```
$ hf-fm clean --older-than 30
  Will remove:
    EleutherAI/pythia-1.4b  (2.80 GiB, last accessed 45 days ago)
  Keep:
    google/gemma-2-2b-it    (accessed 2 days ago)
  Proceed? [y/N] y
  Removed 1 repo, freed 2.80 GiB

$ hf-fm clean --repo EleutherAI/pythia-1.4b --yes
  Removed EleutherAI/pythia-1.4b (2.80 GiB)

$ hf-fm clean
  error: at least one of --older-than, --repo, or --all is required
```

### Flags

| Flag | Description |
|------|-------------|
| `--older-than <DAYS>` | Only remove repos not accessed within this many days |
| `--repo <REPO_ID>` | Remove a specific repo (repeatable) |
| `--all` | Remove all cached repos (use with `--except` to keep specific ones) |
| `--except <REPO_ID>` | Keep a specific repo even if it matches other filters (repeatable) |
| `--yes` | Skip interactive confirmation |

### Implementation

1. Scan cache via `cache_summary()` to list all repos
2. Apply filters: `--older-than` checks file modification times, `--repo` selects specific repos, `--except` excludes repos
3. Display plan (what will be removed, what kept)
4. If not `--yes`: prompt for confirmation via `stdin`
5. For each repo to remove: `std::fs::remove_dir_all(repo_dir)`
6. Report summary: repos removed, bytes freed

### "Last accessed" heuristic

Use the most recent modification time among all files in the repo's snapshot directory. This is an approximation — HF cache doesn't track access times explicitly — but it's consistent with how `huggingface-cli` determines staleness.

### Library API

```rust
/// Result of a cache clean operation.
pub struct CleanResult {
    /// Repos that were removed.
    pub removed: Vec<CleanedRepo>,
    /// Total bytes freed.
    pub bytes_freed: u64,
}

/// A repo that was removed during cleaning.
pub struct CleanedRepo {
    /// The repository identifier.
    pub repo_id: String,
    /// Bytes freed by removing this repo.
    pub bytes: u64,
}

/// Removes cached repositories matching the given criteria.
///
/// At least one of `repos`, `older_than_days`, or `all` must select
/// repos for removal. `except` overrides any selection.
///
/// # Errors
///
/// Returns [`FetchError::Io`] if directory removal fails.
/// Returns [`FetchError::InvalidArgument`] if no selection criteria provided.
pub fn clean_cache(
    repos: Option<&[String]>,
    older_than_days: Option<u64>,
    all: bool,
    except: &[String],
) -> Result<CleanResult, FetchError>;
```

## Implementation Order

| Order | Subcommand | Rationale |
|-------|-----------|-----------|
| 1 | `du` | Simplest — reuses `cache_summary()` almost entirely. Immediate value: see what's taking space. |
| 2 | `verify` | Needs network + async (SHA256 computation). Reuses `checksum.rs` and `repo.rs`. Users need to verify before deciding what to clean. |
| 3 | `clean` | Destructive — implement last, after `du` and `verify` are proven. Needs interactive prompt (stdin). |

## CLI Tests

Following the v0.8.1 pattern, each subcommand gets CLI integration tests in `tests/cli.rs`:

- **`du`**: run against `julien-c/dummy-unknown` (cached), verify output contains repo name, file count, "total"
- **`du <REPO_ID>`**: verify per-file output contains known filenames
- **`verify`**: run against cached repo, verify "SHA256 OK" or "no LFS hash" in output
- **`clean --yes`**: test against a repo that can be re-downloaded (or test the dry-run/confirmation path)
